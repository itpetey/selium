description: "Programming guest modules with the selium-guest crate in the current Selium model."

`selium-guest` is the main crate used inside guest modules compiled for the Selium runtime. In Rust source the package is imported as `selium_guest`, and a typical guest crate exposes one or more `#[selium_guest::entrypoint]` functions.

There is no guest operating system inside the module. Instead, guest code uses `selium_guest` to talk to the host for lifecycle, I/O, networking, storage, time, queue, and shared-memory operations. Most guests start with the crate-root helpers (`entrypoint`, `spawn`, `yield_now`, `shutdown`) and then move into the capability-specific modules they need.

## Module IDs and the work directory

Selium resolves guest modules relative to the runtime work directory. In practice, most local setups keep compiled WASM files under `modules/`:

- `./target/.../my_module.wasm` is your build artifact
- `<work-dir>/modules/my_module.wasm` is where the runtime expects to find it

For CLI and runtime module specs, `my_module.wasm` and `modules/my_module.wasm` both map to the same module ID. The runtime normalizes the `modules/` prefix away before execution.

## Entrypoints

Entrypoints are the guest functions the runtime is allowed to invoke. They are your public process surface.

In Rust, you annotate them with `#[selium_guest::entrypoint]`:

```rust
use std::time::Duration;

#[selium_guest::entrypoint]
pub async fn start() -> Result<(), ()> {
    let worker = selium_guest::spawn(async {
        let _ = selium_guest::time::sleep(Duration::from_millis(10)).await;
    });

    worker.await;
    Ok(())
}
```

That small example matches the crate's root surface: declare an entrypoint, optionally spawn guest tasks, and reach into modules like `time` from there. Entrypoints can also take typed ABI arguments, which is how the same module can expose multiple operational entrypoints such as `start`, `launch`, `reconfigure`, or worker-style child entrypoints started from another guest.

## Arguments

Selium supports scalar, buffer, and resource arguments. In practice you supply them in one of two places:

- a runtime module spec such as `path=...;params=utf8,i32;args=utf8:billing,i32:3`
- `selium_guest::process::ProcessBuilder` from another guest

The current scalar and buffer kinds are:

- Scalars: `i8`, `u8`, `i16`, `u16`, `i32`, `u32`, `i64`, `u64`, `f32`, `f64`
- Buffers: `utf8` (string), `buffer`/`hex` (raw bytes)
- Resources: `resource` (a numeric guest resource handle / resource id)

When `params` is omitted from a module spec, each argument must carry its own type prefix.

## Managed bindings

Selium's current discovery model is built around managed bindings rather than guest-created global registries. When a process is launched with public endpoint bindings, Selium prepends that bindings payload to the entrypoint arguments.

That is why many current examples begin like this:

```rust
#[selium_guest::entrypoint]
pub async fn start(bindings: DataValue) -> Result<()> {
    let mut writer = selium_guest::io::managed_event_writer(&bindings, "echo.requested", 1).await?;
    Ok(())
}
```

`selium_guest::io` accepts either structured `DataValue` bindings or the current encoded byte form, so most new guest code can pass the bindings value directly.

Use these helpers when you want the runtime or control plane to wire public event endpoints for you.

For public services and streams, the generated bindings layer uses `selium_guest::network::Session` handles plus the `selium_guest::network::rpc` and `selium_guest::network::stream` helpers. Reach for `listen(...)` when the runtime has already provisioned an inbound binding, and `connect(...)` / `quic::connect(...)` / `http::connect(...)` when the guest is opening an outbound session.

## ProcessBuilder

Guests can start other Selium processes directly using `selium_guest::process::ProcessBuilder`. This is how supervisor-style modules start worker entrypoints, pass typed arguments, and grant only the capabilities or runtime-managed resources the child needs.

`ProcessBuilder` lets you set:

- the target module ID and entrypoint name
- the ABI signature and typed arguments
- granted capabilities
- network egress profiles and ingress bindings
- storage log and blob grants

Use it for parent/child orchestration inside the runtime. Use the `selium` CLI when you want operational control from outside the runtime.

## Capabilities

Capabilities are Selium's permission system. Any hostcall that touches process lifecycle, time, network, storage, shared memory, or queue I/O must be backed by an explicit grant.

Some common capability names in the current ABI are:

- `session_lifecycle`, `process_lifecycle`, `time_read`
- `shared_memory`, `queue_lifecycle`, `queue_writer`, `queue_reader`
- `network_lifecycle`, `network_connect`, `network_accept`, `network_stream_read`, `network_stream_write`, `network_rpc_client`, `network_rpc_server`
- `storage_lifecycle`, `storage_log_read`, `storage_log_write`, `storage_blob_read`, `storage_blob_write`

Grant as little authority as possible. For direct CLI launches, `selium start --module ...` can build a module spec for you; for lower-level launches, use an explicit module spec or `ProcessBuilder` and enumerate what the process needs.

## Guest API surface

The guest crate keeps its root surface intentionally small and splits the rest by host domain:

- `selium_guest::entrypoint`, `spawn`, `yield_now`, and `shutdown` at the crate root for entrypoints and guest task coordination
- `selium_guest::process`, `session`, and `time` for lifecycle-oriented hostcalls
- `selium_guest::io` for managed public event bindings
- `selium_guest::network` for managed ingress/egress sessions, streams, and RPC
- `selium_guest::storage` for runtime-managed durable logs and blob stores
- `selium_guest::queue` and `selium_guest::shm` for lower-level data-plane control

Most user code should start with the crate-root helpers, `io`, `process`, and the contract-generated bindings from `selium idl compile`, then reach for the lower-level modules only when you need more control.

## Choosing a launch path

- Use `selium start` / `list` / `stop` when you want a fast operational inner loop.
- Use `idl publish`, `deploy`, `connect`, `scale`, and `agent` when you want stable workload identities and desired-state reconciliation.
- Use `ProcessBuilder` when one guest needs to orchestrate another guest from inside the runtime.
