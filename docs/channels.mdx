Current Selium is contract-first. The things you name and bind are not ad-hoc queues or hand-wired channel graphs; they are contract-defined **events**, **services**, and **streams**.

## The three user-facing data shapes

- **Events** are replayable message flows. In the control-plane model, they are the endpoints you wire with `selium connect`.
- **Services** are request/response interfaces with generated typed client and server helpers.
- **Streams** are long-lived byte or typed frame sessions for protocol-aware flows.

In the control-plane API these map to `ContractKind::{Event, Service, Stream}`. In the host/client SDK they map to `ChannelKind::{Event, Rpc, Stream}`.

## Contracts define the public surface

A `.selium` package can declare:

- schemas
- events with partition count, retention, delivery guarantee, and replay enablement
- services with protocol metadata plus buffered or streaming bodies
- streams with typed payload schemas

That contract package is what `selium idl compile` turns into Rust bindings and what `selium idl publish` registers into the control plane.

## Retention and replay

Replay is a first-class part of the current data plane.

- retention is configured with `RetentionPolicy { max_entries }`
- subscriptions can start from `ReplayStart::Earliest`, `Latest`, `Sequence`, `Timestamp`, or `Checkpoint`
- host-side tools can backfill data with `selium-sdk-rust` before they switch over to live subscription

That makes event streams useful for both normal live delivery and operator or test-time reprocessing.

## Managed bindings vs lower-level primitives

Most users should start with the managed surfaces:

- `selium start --event-reader` / `--event-writer` for direct operational launches
- control-plane-managed endpoint bindings for workload-to-workload event routing
- generated service and stream helpers layered on `selium_guest::network`

If you need lower-level control inside a guest, `selium_guest::queue` and `selium_guest::shm` are still there. They are the escape hatch, not the default mental model.

## Pick the right layer

- Use [Guest API](./applications) when you are writing a WASM guest.
- Use [Rust SDK](./rust-sdk) when you need host-side publish/subscribe or replay.
- Use [Networking](./networking) when the contract surface is a service or stream rather than an event.
- Use [Examples](./examples) when you want the quickest reference implementation.
