Selium is a developer-first platform for running WebAssembly workloads as isolated processes under a single runtime daemon. The current platform is organized around four surfaces:

- `selium-runtime` runs the local daemon, owns the work directory, terminates mTLS, and boots the control plane.
- `selium` is the operational and control-plane CLI: `start`, `stop`, `list`, `deploy`, `connect`, `scale`, `observe`, `replay`, `nodes`, `agent`, and `idl`.
- `selium-guest` (`selium_guest` in Rust code) is the main guest-side crate: entrypoints, executor helpers, and guest hostcalls for I/O, process orchestration, network, storage, queue, shared memory, session, and time.
- `selium-sdk-rust` (`selium_sdk_rust` in Rust code) is the host/client-side SDK for typed or byte-level publish/subscribe and replay outside a guest module.

Selium still has the same core promise: strong isolation, explicit authority, and composable APIs. What changed is the mental model. The current workflow is contract-first and daemon-first.

If your Rust code compiles into a Selium guest WASM module, start with `selium-guest`. If it runs alongside or outside the runtime, start with `selium-sdk-rust` instead. The shared ABI layer behind both stories is `selium-abi`, which defines guest-facing value types plus the `rkyv`-oriented encode/decode helpers used across guest and host Rust APIs.

## The current mental model

1. **Define contracts** for the public events, services, and streams your workload exposes.
2. **Build guest modules** that expose one or more `#[selium_guest::entrypoint]` functions.
3. **Run a daemon** with a work directory, certificates, and a node identity.
4. **Choose an execution path**:
   - use `selium start` / `list` / `stop` for direct operational launches
   - use `idl publish`, `deploy`, `connect`, `scale`, and `agent` for desired-state control-plane workflows

That split is intentional. Direct `start` is great for local iteration and one-off workloads. The control plane is how you name workloads, wire public endpoints, scale replicas, and reconcile onto nodes.

## What Selium applications look like now

Applications are plain WASM modules with entrypoints. Selium passes typed arguments across the ABI boundary through `selium-abi`, and public endpoint launches also prepend a bindings payload that guests use directly for managed event readers and writers. Service and stream workloads use generated helpers over `selium_guest::network`.

In practice, most guest modules use a small set of building blocks:

- the crate-root `#[selium_guest::entrypoint]` macro plus `spawn`, `yield_now`, and `shutdown`
- `selium_guest::io` for managed public event bindings
- `selium_guest::process::ProcessBuilder` to start other Selium processes
- `selium_guest::network` for daemon-managed ingress and egress plus generated service/stream sessions
- `selium_guest::storage`, `queue`, `shm`, `session`, and `time` for stateful and systems-level flows

## Two workflows you should keep separate

### Direct runtime operations

Use these when you want a fast inner loop:

- build a module to `wasm32-unknown-unknown`
- copy it into the runtime work directory
- run `selium start --node ... --replica-key ... --module ...`
- inspect with `selium list`
- stop it with `selium stop`

### Control-plane managed workloads

Use these when you want stable identities and wiring between workloads:

- publish contracts with `selium idl publish`
- declare workloads with `selium deploy`
- connect public event endpoints with `selium connect`
- reconcile desired state with `selium agent --once` or the long-running agent loop
- inspect state with `selium observe`, `selium replay`, and `selium nodes`
