`selium` is the operator-facing CLI for Selium. Most commands talk to a running `selium-runtime` daemon, while `idl compile` is local and generates bindings directly from a `.selium` package.

## Shared connection flags

Daemon-backed commands share the same connection tuple:

- `--daemon-addr`
- `--daemon-server-name`
- `--ca-cert`
- `--client-cert`
- `--client-key`

You can pass those inline, or load them from a CLI config file with `-c/--config`. `idl compile` does not require a daemon connection.

## Direct operational commands

Use these when you want a tight local loop against a known node:

- `start` launches one replica on `--node` with a required `--replica-key`
- `list` shows running replicas, optionally filtered to one node
- `stop` stops one running replica by `--node` and `--replica-key`

`start` accepts either:

- `--module modules/my_workload.wasm` for the common path, or
- `--module-spec 'path=...;entrypoint=...;capabilities=...;args=...'` when you need the full launch surface

It can also inject managed public event bindings with repeated `--event-reader` and `--event-writer` flags.

## Control-plane commands

Use these when you want stable names and desired-state workflows:

- `idl compile` generates Rust bindings from a `.selium` package
- `idl publish` registers that package with the control plane
- `deploy` declares a workload, module, replica count, isolation, and one or more `--contract` bindings
- `connect` creates a pipeline edge between two workloads for one contract-defined event endpoint
- `scale` updates the replica count for an existing workload
- `agent` reconciles desired state onto a node, once or on an interval

The read-side commands are:

- `discover` for workload, endpoint, and running-process discovery with `--json` output
- `nodes` for node inventory and liveness
- `observe` for the current control-plane view
- `replay` for recent control-plane events, with cursor and attribution filters plus `--json` output
- `inventory` for attributed workload, module, pipeline, and node inventory with `--json` output
- `usage` for attributed runtime usage export from a specific node daemon with `--json` output
- `attach tenant/namespace/workload#event:stdout` (or `stderr` / `std*`) for live guest logs

## External-consumer adoption cookbook

Use the read-side export commands with `--json` when Selium needs to hand stable, machine-consumable inventory, replay, and usage data to an external consumer. The already-tested adoption loops are:

1. **inventory -> replay** for a control-plane bootstrap plus delta catch-up
2. **discover -> nodes -> replay/usage** for a live handoff onto one concrete workload and one live daemon

If the downstream consumer is a host-side Rust process, keep [Rust SDK](./rust-sdk) nearby for payload decoding and metadata-preserving replay helpers. The CLI remains the authoritative source for the control-plane and daemon command flow.

### 1. Bootstrap with inventory, then continue from replay deltas

Start with inventory when you need one bounded snapshot of workloads, modules, pipelines, and nodes before switching to replay:

```bash
selium inventory \
  --external-account-ref acct-123 \
  --workload tenant-a/media/ingest \
  --json
```

The JSON response contains:

- `snapshot_marker.last_applied`: the control-plane snapshot position reflected by this inventory export
- `workloads`, `modules`, `pipelines`, and `nodes`: filtered arrays exposing Selium-native identifiers plus any attached `external_account_ref` values

For the bootstrap-to-replay handoff, continue from the next durable sequence after that marker. For example, if `snapshot_marker.last_applied` is `41`, resume with `selium replay --since-sequence 42 ... --json` so downstream consumers only ingest mutations that landed after the exported inventory snapshot.

### 2. Discover workloads, endpoints, and live usage-export nodes

Use `discover` when an external consumer needs to find Selium-native workload and endpoint identities without prior repository knowledge:

```bash
selium discover \
  --workload-prefix tenant-a/media/ \
  --json
```

The discovery JSON contains `workloads` plus contract-bound `endpoints`, all keyed by Selium-native workload and public-endpoint identity.

When you need to resolve one concrete target, use one of the explicit resolve flags:

```bash
selium discover \
  --resolve-endpoint tenant-a/media/router#service:camera.detect \
  --workload tenant-a/media/router \
  --json
```

That response resolves the selected workload, endpoint, or running process using the control plane's existing discovery state. Once you know the workload and endpoint you care about, list live nodes that can serve runtime usage export:

```bash
selium nodes --max-staleness-ms 20000 --json
```

The node JSON includes `name`, `daemon_addr`, `daemon_server_name`, `live`, and `supported_isolation`. Use `name` as the `selium usage --node ...` selector when collecting attributed runtime usage from a live daemon.

#### Cookbook handoff: discover -> nodes -> first bounded replay/usage pull

Use this as the default external-consumer handoff when you need to move from Selium discovery to the first machine-consumable export batch:

1. Discover candidate workloads and public endpoints for the tenant or namespace you care about.
2. Resolve one concrete endpoint so you can pin the exact Selium-native workload key you will join against downstream.
3. List live nodes and select the daemon that will serve runtime usage export.
4. Start with a bounded replay pull for control-plane audit history.
5. Start with a bounded usage pull from the selected live node, then continue from the returned `next_sequence`.

```bash
selium discover \
  --workload-prefix tenant-a/analytics/topology- \
  --json

selium discover \
  --resolve-endpoint tenant-a/analytics/topology-ingress#event:ingest.frames \
  --json

selium nodes --json

selium replay \
  --workload-key tenant-a/analytics/topology-ingress \
  --limit 1 \
  --json

selium usage \
  --node node-a \
  --workload tenant-a/analytics/topology-ingress \
  --limit 1 \
  --json
```

Keep the first replay/usage call deliberately small (`--limit 1` or another low bound) so downstream consumers can validate identity joins before widening the window. `replay` follows the discovered workload key, while `usage` must target one live daemon from `selium nodes --json` plus the workload key you resolved during discovery. If you are bootstrapping against a slower local or staging cluster, set `--max-staleness-ms` high enough that recent heartbeats still count as live.

### 3. Continue replay incrementally

Use `replay` when an external consumer needs durable mutation history together with a replay cursor:

```bash
selium replay \
  --since-sequence 41 \
  --external-account-ref acct-123 \
  --workload-key tenant-a/media/ingest \
  --module ingest.wasm \
  --pipeline tenant-a/media/camera \
  --node node-a \
  --limit 25 \
  --json
```

The JSON response includes:

- `start_sequence`: the durable replay point used for this batch
- `next_sequence`: the cursor you can pass back on the next replay call
- `high_watermark`: the newest durable replay sequence available at query time
- `events`: the filtered replay events, each still carrying Selium-native resource identity so downstream systems can join them back to deployments, workloads, modules, and pipelines without Selium owning the external account model

### 4. Continue usage export incrementally

Use `usage` when an external consumer needs immutable sampled runtime usage plus the next durable resume cursor:

```bash
selium usage \
  --node node-a \
  --since-sequence 41 \
  --external-account-ref acct-123 \
  --workload tenant-a/media/ingest \
  --module ingest.wasm \
  --window-start-ms 1000 \
  --window-end-ms 2000 \
  --limit 25 \
  --json
```

The JSON response includes `records`, `next_sequence`, and `high_watermark`. Each usage record carries both Selium-native identity (`sample.workload_key`, `sample.attribution.module_id`) and the opaque join key (`sample.attribution.external_account_ref`) so downstream systems can meter, rate, or bill without Selium modelling external account lifecycle state.

### Troubleshooting appendix

- **`inventory` is missing the workload or `external_account_ref` you expected**: confirm you are filtering with the exact control-plane workload key (`tenant/namespace/workload`) and the same opaque account key attached to the deployment or pipeline. The bootstrap flow only sees resources already committed to the control plane.
- **The replay handoff repeats snapshot-time data**: resume from `snapshot_marker.last_applied + 1`, not from the marker itself. The inventory bootstrap smoke test uses this exact cursor handoff to avoid replaying pre-snapshot mutations.
- **`replay` returns no events even though the command succeeds**: check `next_sequence` and `high_watermark` in the JSON response. An empty batch can still be correct when no post-cursor mutations match the filters you supplied.
- **`discover --resolve-endpoint ...` does not pin the workload you meant**: use the full endpoint key shape `tenant/namespace/workload#kind:name`, then carry the returned workload key forward into `replay` and `usage`.
- **`usage` cannot find records**: pick a `live` node from `selium nodes --json` and pass its `name` to `selium usage --node ...`. Unlike control-plane replay, runtime usage export is daemon-scoped and must target one live node.
- **`nodes` shows no live daemon in local or staging environments**: increase `--max-staleness-ms` so recent heartbeats still count as live before you choose the usage-export target.

## Guest log attach

`attach` is currently scoped to workload guest log event selectors only:

- `tenant/namespace/workload#event:stdout`
- `tenant/namespace/workload#event:stderr`
- `tenant/namespace/workload#event:std*`

When a selector resolves to multiple guest log streams, each output line is prefixed with the stream name so `stdout` and `stderr` stay distinguishable.

## Know which identity you are holding

Selium exposes two naming layers on purpose:

- **user-facing workload identity**: `tenant/namespace/workload`
- **operational running replica identity**: the `--replica-key` you use with `start` / `stop`

Control-plane commands work with workload and endpoint identities. Operational commands work with node-local running replicas.

## A good default split

- Start with `idl publish` when the workload is contract-defined.
- Use `start` for direct local launches and example iteration.
- Use `deploy` + `connect` + `agent --once` when you want a real multi-workload topology.
