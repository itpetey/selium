---
title: "Networking"
description: "Protocol-aware guest networking with runtime-managed ingress and egress."
---

Selium's current networking surface is runtime-managed and protocol-aware. Guests do not bind raw sockets directly. Instead, the runtime defines named ingress bindings and egress profiles, and guest code receives access to those named resources.

## What exists today

- Protocols: `NetworkProtocol::{Quic, Http}`
- Interaction kinds: `InteractionKind::{Stream, Rpc}`

In practice that means two main patterns today:

- QUIC stream sessions for long-lived bidirectional traffic
- HTTP-over-TLS request/response RPC

The public examples are [network-quic-stream](https://github.com/itpetey/selium/tree/main/examples/network-quic-stream) and [network-http-rpc](https://github.com/itpetey/selium/tree/main/examples/network-http-rpc).

## Runtime-owned configuration

The daemon config owns the network surface:

- **egress profiles** declare the protocol, allowed interaction kinds, allowed authorities, and client TLS material
- **ingress bindings** declare the protocol, allowed interaction kinds, listen address, and server certificate/key

Guests and child processes only see those resources by name. Access is granted through a runtime module spec or through `selium_guest::process::ProcessBuilder`.

## Guest API

The core guest entrypoints are straightforward:

- `selium_guest::network::listen(binding_name)` to accept inbound sessions
- `selium_guest::network::quic::connect(profile, authority)` for outbound QUIC
- `selium_guest::network::http::connect(profile, authority)` for outbound HTTP-over-TLS

Once connected, the guest works with `Listener`, `Session`, `StreamChannel`, and RPC exchange helpers:

- `Listener::accept(...)` accepts a session
- `network::stream::open(...)` or generated stream helpers create outbound stream channels
- generated RPC helpers accept requests, read buffered or streaming bodies, and send typed responses

Authorities are passed in `host:port@server-name` form, as shown in the network examples.

## Contracts and generated helpers

Service and stream contracts are where the current user-facing ergonomics live.

- `selium idl compile` generates typed helpers for client/server or stream usage
- generated service helpers sit on top of `selium_guest::network` RPC sessions
- generated stream helpers sit on top of `selium_guest::network` stream channels

That is why the current networking docs should be read together with [Control Plane](./control-plane) and [Guest API](./applications), not as a separate legacy subsystem.

## Keep the responsibility split clear

- Use `selium_guest::io` for managed **event** bindings
- Use `selium_guest::network` for **service** and **stream** workloads
- Use [Runtime](./runtime) when you need to define the binding/profile names the guest code will consume

Selium is still early here, but the direction is already clear: runtime-defined network resources, guest access by name, and typed helpers generated from contracts rather than hand-rolled protocol plumbing.
