`selium-sdk-rust` is the host-side Rust SDK for Selium publish, subscribe, and replay flows. Use it when your code is **not** running as a Selium WASM guest but still needs to interact with Selium channels.

Typical users include tests, sidecars, adapters, ingest tools, and operator utilities. If your code is running inside a Selium guest module, use [Guest API](./applications) and the `selium-guest` crate instead.

## Start with a Context

The SDK centers on `Context`, the host/client entry point for creating channels, publishing messages, subscribing, and replaying retained frames:

- `Context::new()` for the default in-memory setup
- `Context::builder().with_core_io(...)` when you want to inject a specific `CoreIo`
- `runtime_settings()` when you want to inspect the current runtime metadata on the context
- `set_runtime_encoding(...)` when the runtime needs to enforce a particular encoding policy

The default runtime metadata records `rkyv`, which matches the current contract and guest tooling story. Typed publishers and subscribers in this crate still encode payloads as `rkyv` today even if you update the stored setting with `set_runtime_encoding(...)`.

## Quick example

```rust
use rkyv::{Archive, Deserialize, Serialize};
use selium_abi::decode_rkyv;
use selium_io_core::ChannelKind;
use selium_io_durability::{ReplayStart, RetentionPolicy};
use selium_sdk_rust::Context;

#[derive(Debug, Clone, PartialEq, Eq, Archive, Serialize, Deserialize)]
#[rkyv(bytecheck())]
struct DemoEvent {
    id: u64,
}

let context = Context::new();
context.ensure_channel("demo.events", ChannelKind::Event, RetentionPolicy::default())?;

context
    .publisher::<DemoEvent>("demo.events")
    .publish(DemoEvent { id: 7 })?;

let replay = context.replay_bytes("demo.events", ReplayStart::Earliest, 1)?;
let event: DemoEvent = decode_rkyv(&replay[0])?;
assert_eq!(event.id, 7);
# Ok::<(), Box<dyn std::error::Error>>(())
```

## Create or reuse channels

The SDK exposes channel lifecycle directly:

- `create_channel(name, ChannelKind, RetentionPolicy)`
- `ensure_channel(name, ChannelKind, RetentionPolicy)`

The current channel kinds are `Event`, `Rpc`, and `Stream`. Retention is configured with `RetentionPolicy { max_entries }`, and `ensure_channel(...)` also validates that an existing channel still has the requested kind.

## Typed and byte-oriented publish/subscribe

You can work at two levels:

- `publisher<T>` and `subscriber<T>(..., ReplayStart::...)` for typed `rkyv` payloads
- `byte_publisher` and `byte_subscriber(..., ReplayStart::...)` for raw byte frames

Typed publishers automatically encode payloads as `application/rkyv`. Typed subscribers decode them back into Rust types on receipt. Use the byte-oriented API when you already have encoded frames or need interoperability with non-Rust producers and consumers.

When you need durable metadata as well as the payload, use `Context::replay_frames(...)` and `ByteSubscriber::recv_frame()`. These preserve the frame `sequence`, `timestamp_ms`, `headers`, `channel`, and `payload`, while `replay_bytes(...)` and `ByteSubscriber::recv()` remain available for payload-only compatibility.

## Replay and backfill

Replay is part of the public SDK surface.

- `subscriber(..., ReplayStart::Earliest | Latest | Sequence | Timestamp | Checkpoint)` lets you start from the right point in retained channel history
- `Context::replay_frames(channel, start, limit)` preserves frame headers and other durable metadata for event/log consumers
- `replay_bytes(channel, start, limit)` gives host-side tools a simple backfill path

That makes the SDK useful for tests, ingest tools, sidecars, diagnostics, and adapters that need to bootstrap from history before switching to live traffic.

### Metadata-preserving replay example

```rust
use selium_io_durability::ReplayStart;
use selium_sdk_rust::Context;

let context = Context::new();
let replay = context.replay_frames("demo.events", ReplayStart::Earliest, 1)?;

let frame = &replay[0];
let external_account_ref = frame.headers.get("external_account_ref");

assert_eq!(external_account_ref, Some(&"acct-ext-123".to_string()));
assert_eq!(frame.payload, b"raw event bytes".to_vec());
# Ok::<(), Box<dyn std::error::Error>>(())
```

For external-consumer handoff code, treat `Frame` as the durable envelope: `frame.sequence` is your replay cursor, `frame.timestamp_ms` is your event time, `frame.headers.get("external_account_ref")` preserves the opaque join key, and `frame.payload` remains the raw domain bytes you decode on your side.

### External-consumer replay → live handoff cookbook

Use the same durable cursor for both phases:

1. Load the last committed `next_sequence` from your own checkpoint store (or start at `1` for a fresh consumer).
2. Call `Context::replay_frames(...)` with `ReplayStart::Sequence(next_sequence)` to backfill retained frames.
3. Process each `Frame`, then advance `next_sequence = frame.sequence + 1` after your handler succeeds.
4. Switch to `byte_subscriber(..., ReplayStart::Sequence(next_sequence))` and keep consuming with `recv_frame()` so the live path preserves the same metadata envelope.

```rust
use selium_io_durability::ReplayStart;
use selium_sdk_rust::Context;

async fn consume_external_feed(
    context: &Context,
    mut next_sequence: u64,
) -> Result<(), Box<dyn std::error::Error>> {
    let replay = context.replay_frames(
        "usage.events",
        ReplayStart::Sequence(next_sequence),
        256,
    )?;

    for frame in &replay {
        let external_account_ref = frame.headers.get("external_account_ref");
        let workload_key = frame.headers.get("workload_key");
        let _ = (external_account_ref, workload_key, &frame.payload);
        next_sequence = frame.sequence + 1;
    }

    let mut subscriber = context.byte_subscriber(
        "usage.events",
        ReplayStart::Sequence(next_sequence),
    )?;

    loop {
        let frame = subscriber.recv_frame().await?;
        let _ = frame.headers.get("external_account_ref");
        next_sequence = frame.sequence + 1;
        // Persist `next_sequence` after your application commits the frame.
    }
}
```

This recipe stays entirely on the existing SDK surface: `replay_frames(...)` gives you the durable backlog, `ReplayStart::Sequence(...)` is the stored resume cursor, and `recv_frame()` preserves `headers`, `timestamp_ms`, and `payload` on the live path.

## When to use the SDK vs the guest crate

- Use `selium-sdk-rust` for host-side or external Rust processes that need channels, publish/subscribe, or replay.
- Use [Guest API](./applications) when the code is running inside a Selium WASM guest and needs guest hostcalls or managed bindings.
- Use [Data Plane](./channels) when you want the broader mental model for events, services, streams, retention, and replay.
