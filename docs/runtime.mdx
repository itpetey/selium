The Selium runtime (`selium-runtime`) is the host process behind everything else in the platform. It owns the work directory, hosts the kernel services, terminates mTLS for the CLI and peer traffic, and in daemon mode boots first-party system modules through a shared lifecycle contract. Today the control-plane guest is the first module on that path and powers workload, pipeline, and node operations.

## Work directory

The runtime needs one base directory for code, certificates, and durable state. That is `--work-dir` (or `SELIUM_WORK_DIR`).

- `certs/` for the CA, server certificate, client certificate, and keys
- `modules/` for guest WASM modules
- additional state paths such as the control-plane state directory and runtime-managed storage locations

Most local setups begin by generating a certificate bundle:

```bash
selium-runtime generate-certs --output-dir .selium-local/certs
```

## Daemon mode

For day-to-day development, daemon mode is the primary entrypoint:

```bash
selium-runtime \
  --work-dir .selium-local \
  daemon \
  --listen 127.0.0.1:7100 \
  --cp-node-id local-node \
  --cp-public-addr 127.0.0.1:7100 \
  --cp-state-dir control-plane \
  --quic-ca certs/ca.crt \
  --quic-cert certs/server.crt \
  --quic-key certs/server.key \
  --quic-peer-cert certs/client.crt \
  --quic-peer-key certs/client.key
```

The important daemon flags are:

- `--listen` for the CLI and peer QUIC endpoint
- `--cp-node-id` for the node's logical control-plane identity
- `--cp-peer` for additional node peers in multi-node setups
- `--cp-public-addr` and `--cp-server-name` for the address and TLS name advertised into node discovery
- `--cp-state-dir` for raft state, snapshots, and durable control-plane events
- `--quic-ca`, `--quic-cert`, `--quic-key`, `--quic-peer-cert`, and `--quic-peer-key` for mTLS material

When daemon mode starts, Selium boots its control-plane guest through the system-module governance path and exposes the lifecycle/control API that the `selium` CLI uses.

That split is intentional:

- the host keeps permanent substrate responsibilities: execution, isolation, transport termination, authentication and authorisation enforcement, capability grants, and runtime-managed bridge or process realization
- first-party system modules keep guest-owned policy: control API meaning, desired-state interpretation, bridge topology and lifecycle policy, and workload orchestration intent

The current control-plane module remains the proving ground for this model. The daemon is expected to behave like a trusted gateway and realization engine, not a second semantic control plane.

That means three concrete boundaries:

- control queries and mutations arrive at the daemon with mTLS identity, but the daemon forwards guest-facing authorisation scope to the control-plane guest instead of rewriting desired-state meaning locally
- endpoint bridges are realized from guest-authored bridge intent that names topology, classification, and lifecycle policy; the daemon keeps only the worker and transport state needed to execute that intent
- process lifecycle calls are realized from guest-authored orchestration intent that names the desired transition, reason, and runtime resource posture; the daemon turns that into generic process and capability operations

## Direct module startup

Outside daemon mode, `selium-runtime` can also be used as a direct module launcher with repeated `--module` specs. That is a lower-level path than `selium start`, but it is still useful when you want the runtime to launch one or more modules immediately.

Each `--module` value is a `;`-delimited `key=value` string.

Required keys:

- `path`: relative module path, usually under `modules/`
- `capabilities`: comma-separated [capabilities](./applications#capabilities)

Optional keys:

- `entrypoint`: defaults to `start`
- `network-egress-profiles`
- `network-ingress-bindings`
- `storage-logs`
- `storage-blobs`
- `adaptor`: currently `wasmtime` is the executable path in this build
- `profile`: `standard`, `hardened`, or `microvm`
- `params` / `args` for typed entrypoint arguments

```bash
selium-runtime \
  --work-dir .selium-local \
  --module 'path=modules/typed_entrypoints.wasm;entrypoint=launch;capabilities=session_lifecycle,process_lifecycle,time_read,shared_memory,queue_lifecycle,queue_writer,queue_reader;params=utf8,i32,utf8;args=utf8:billing,i32:3,utf8:blue-green;adaptor=wasmtime;profile=standard'
```

If `params` is omitted, each `args` item must carry its own type prefix such as `utf8:billing` or `i32:3`.

## Runtime-managed network and storage

The runtime also loads network and storage definitions from config:

- network **egress profiles** define allowed outbound authorities plus client TLS material
- network **ingress bindings** define listener names, protocols, interactions, and server cert/key paths
- storage **logs** and **blob stores** define durable state the runtime can grant into workloads

Those resources are then referenced by name from module specs or guest `ProcessBuilder` calls.

## The CLI contract with the daemon

The `selium` CLI connects to the daemon over QUIC using these connection settings:

- `--daemon-addr`
- `--daemon-server-name`
- `--ca-cert`
- `--client-cert`
- `--client-key`

That connection is what powers `start`, `stop`, `list`, `deploy`, `connect`, `scale`, `observe`, `replay`, `nodes`, `agent`, and `idl publish`.

Use `--log-format text|json` (or `SELIUM_LOG_FORMAT`) when you want to change daemon log output formatting.
